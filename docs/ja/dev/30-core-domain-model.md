# A2O 中核ドメインモデル

この文書は、A2O の中核ドメインオブジェクトと責務境界を定義する。

読む目的は、A2O がカンバンの表示状態やログではなく、ドメイン状態をもとに次の行動を決めることを理解することである。タスク、実行、フェーズ、証跡をどこで保持し、どの状態遷移を許すかを判断するときに読む。外部システムとの接続は [95-kanban-adapter-boundary.md](95-kanban-adapter-boundary.md)、エージェント実行は [70-agent-worker-gateway-design.md](70-agent-worker-gateway-design.md) を参照する。

## ランタイムの流れ上の位置づけ

この文書は、スケジューラがカンバンタスクを処理対象として選んだ後、Engine がタスクライフサイクルをどう表現するかを扱う。タスク、実行、フェーズ、スコープスナップショット、ソース記述子、証跡は、a2o-agent へジョブを渡す前後の判断と、結果をカンバン / 証跡に戻す判断の正本である。

## モデル原則

- ドメインオブジェクトは原則として不変にする。
- ドメインオブジェクトはプロダクト固有のファイルパスやコマンドを知らない。
- インフラ層は外部システムをドメインオブジェクトへ変換する。
- アプリケーションサービスは状態遷移を隠さずにドメインルールを進行させる。
- 公開診断は、一時ログの再構成ではなくドメイン状態から導出する。

## タスク

`Task` は中心となる集約ルートである。

タスクが持つもの:

- `ref`
- `kind`
- `status`
- `current_run_ref`
- `parent_ref`
- 編集範囲
- 検証範囲

タスク状態はカンバンレーンそのものではなく、内部スケジューラ状態である。A2O はアダプター境界でカンバンレーンと内部状態を相互変換する。

## 実行

`Run` は 1 つのタスクフェーズを 1 回実行する試行を表す。

実行が持つもの:

- 実行参照
- タスク参照
- フェーズ
- ワークスペース種別
- ソース記述子
- スコープスナップショット
- 成果物の所有者
- 状態
- 終了結果
- ブロック診断の要約

1 つのタスクは複数の実行を持てる。現在のタスク状態は、実行中の実行または最新の関連実行を指す。

## フェーズ

フェーズは実行中の作業種別を表す。

- implementation
- review
- parent_review
- verification
- remediation
- merge

フェーズ遷移はドメインルールである。CLI 条件分岐の各所へ分散して埋め込んではならない。

## スコープスナップショット

スコープスナップショットは実行の編集範囲と検証範囲を固定する。これにより、実行記録後のカンバンラベルやパッケージ変更で、記録済み証跡の意味が変わらないようにする。

## ソース記述子

ソース記述子は、実行で使ったソースの由来を記録する。

- ブランチの先頭
- 分離されたコミット
- 親タスク統合参照
- 本流ターゲット参照

ワークスペース具体化と証跡確認は、どちらもこの記述子に依存する。

## 成果物の所有者

成果物の所有者は証跡の所有者を表す。

- 単独タスク / 子タスクの、タスクが所有する証跡
- 統合処理の、親タスクが所有する証跡

成果物の所有者はスナップショットバージョンを含み、証跡をソース状態に結びつける。

## ブロック診断

ブロック診断は、運用者の対応のための構造化された要約である。失敗は次のような分類に分ける。

- 設定不備
- 未整理のワークスペース
- 実行コマンド失敗
- 検証失敗
- マージ競合
- マージ失敗
- ランタイム失敗

## リポジトリ

ドメインリポジトリは次の永続化契約を提供する。

- タスク
- 実行
- スケジューラ状態
- スケジューラサイクル
- 証跡とブロック診断の参照用モデル

JSON と SQLite は、同じリポジトリ契約の背後にあるインフラ実装の選択肢である。
