# A2O 参照用プロダクト群

この文書は、A2O のコア検証で使う管理対象サンプルプロダクトを定義する。

読む目的は、A2O の変更を実プロダクトだけで検証しないための基準を持つことである。参照用プロダクト群は、単一リポジトリ、複数リポジトリ、親子タスク、検証、マージを小さく再現できる環境であり、リファクタリングや公開仕様変更の安全確認に使う。

## ランタイムの流れ上の位置づけ

この文書は、A2O のランタイムの流れが特定プロダクトに依存せず動くことを検証する参照用プロダクトを定義する。ワークスペース、エージェント境界、検証、マージ、親子タスクの流れ、診断の変更は、少なくとも 1 つの参照用プロダクトで再現・検証できる状態にする。

## 目的

A2O は 1 つの技術スタックに依存せず、一般的なプロダクト構成で動作する必要がある。参照用プロダクト群は、ランタイム、エージェント、カンバン、検証、マージ、親子タスクの流れを検証する、小さくレビューしやすいプロジェクトを A2O に提供する。

## 構成

| プロダクト | パス | 目的 |
|---|---|---|
| TypeScript API / Web | `reference-products/typescript-api-web/` | 1 リポジトリ内の API とブラウザー UI |
| Go API / CLI | `reference-products/go-api-cli/` | 1 つの Go モジュール内のサーバーと CLI |
| Python サービス | `reference-products/python-service/` | 軽量サービスと Python 検証 |
| Java Spring multi-module | `reference-products/java-spring-multi-module/` | Maven reactor 内の `utility-lib`、Spring Boot `web-app`、専用 docs slot |
| 複数リポジトリのフィクスチャ | `reference-products/multi-repo-fixture/` | 親子タスクと複数リポジトリの検証 |

各プロダクトはパッケージを `project-package/` に置く。

## パッケージの契約

各パッケージは次を含む。

- `README.md`
- `project.yaml`
- `commands/`
- `skills/`
- `task-templates/`

パッケージは、結果が決まっているテストまたはビルドコマンド、エージェントの前提条件、編集可能なソース境界、カンバンボードに置けるタスクテンプレートを少なくとも 1 つ定義する。

`project.yaml` は単一の作成者向けパッケージ設定ファイルである。パッケージ識別子、カンバン選択条件、リポジトリスロット、エージェントの前提条件、ランタイム公開コマンド、マージ既定値を持つ。

docs-impact を検証する参照パッケージは、`project.yaml` に `docs` 設定も宣言する。Java Spring multi-module package を docs-impact の標準参照とし、`app`、`lib`、`docs` repo slot、feature docs、shared specs、interface docs、authority source、managed index block、mirror policy を持たせる。

## 検証境界

コア検証は参照用プロダクト群から始める。ランタイム、ワークスペース、ワーカー境界、検証、マージ、パッケージプリセットの変更が少なくとも 1 つの参照用プロダクトで検証できない場合、外部プロダクトの証跡に頼る前に参照用タスクテンプレートを追加または改善するチケットを作成する。

プロダクト群の改善中に外部仕様変更が見つかった場合は、実装前に owner と協議する。

## リリース検証の範囲

プロダクト群は次のリリース検証対象である。

- 単一リポジトリの実装 / 検証 / マージ
- 複数リポジトリの子タスク実装と検証
- 子タスクから親タスクへのマージ
- 親タスクレビューと検証
- 親タスクから本流へのマージ
- ランタイム要約表示とタスク診断
- 証跡の保持
- docs-impact context、evidence、summary、docs repo slot validation

検証実行では、モデルによる揺らぎを切り離すため、結果が決まっているワーカーを使ってよい。ただし検証する公開面は実物のままにする。対象はカンバンの取り込みと遷移、ブランチ名前空間の作成、ワークスペース具体化、ワーカー境界の通信、エージェント側の公開、検証コマンド、マージ、証跡保持である。
